Abstract

Four top-level registration trees currently exist within the Internet 
Assigned Numbers Authority (IANA) as established by RFC 2048 [1]. These 
trees are the 
1) IETF Tree for registrations provided via RFCs, 
2) Vendor Tree for vendor-initiated registrations,
3) Personal or Vanity tree for experimental registrations, and the
4)  ''Special X. Tree'' for unregistered values. 
The MIME Registration Procedures, as outlined in Section 2.1.5 of RFC 
2048, has also established a provision for IANA to create a new top-
level registration tree, with the consent of the IESG, for the rare 
instances of registrations not fitting into one of these four 
categories.

This document establishes a new top-level registration tree, the Codec 
Tree. This tree exists to incorporate codec identifiers within the IANA 


Fleischman                                                      [Page 1]

Internet-Draft                                           January 7, 1998

namespace for codecs that have been already registered by an 
alternative (non-Internet) registry agency. Incorporating these 
identifications within the Internet namespace permits Internet 
protocols, e.g., RTP [2], to refer to these codecs in a standard 
fashion.

The Codec Tree will initially be populated by codec identifiers that 
have already been registered for WAVE and AVI files into the IANA 
namespace. The initial registrar for WAVE and AVI codecs is Microsoft. 
The tree may be subsequently populated by codec identifications coming 
from additional registries (e.g., QuickTime). Specifically, this 
document establishes that the already registered codecs will maintain 
their current registered names when they populate the Codec Tree. For 
example, a video codec with the FourCC identifier value of ''SCRN'' would 
be known as ''video/codec-SCRN'' within the Codec namespace and an audio 
codec with the WAVE Format ID value of 123 would be known as 
''audio/codec-123'' in the Codec namespace. The identities of the codec 
IDs that are taken from the Microsoft Codec Registry are given in 
Appendix A (for audio codecs) and Appendix B (for video codecs).

1 Introduction

Provision exists (e.g., [3]) for codecs defined within IANA's IETF Tree 
to be referenced by Internet Protocols such as RTP [2]. This facility 
has proven to be quite useful for a variety of Internet applications, 
including Video Conferencing (e.g., MBone). A growing desire exists to 
supplement this existing system with a mechanism by which existing 
multimedia content, created outside of distinctly Internet contexts, 
may be integrated into emerging Internet usage scenarios (e.g., for 
RTSP [4]). For this to occur, a mechanism must exist for Internet 
protocols (e.g., [2], [4], [5]) to identify the codecs by which this 
"traditional" multimedia content has been encoded, preferably within 
the IANA namespace.

Several distinct encoding systems exist for identifying the codecs that 
have been used to encode the vast amount of "traditional" multimedia 
content. Each system has its own registry to ensure unique and stable 
codec identifications within that system. Perhaps the best known of 
these registries are Microsoft (for WAVE and AVI content) and Apple 
(for QuickTime content).

Four top-level registration trees currently exist within the Internet 
Assigned Numbers Authority (IANA) as established by RFC 2048 [1]. These 
trees are the 
1) IETF Tree for registrations provided via RFCs, 
2) Vendor Tree for vendor-initiated registrations,


Fleischman                                                      [Page 2]

Internet-Draft                                           January 7, 1998

3) Personal or Vanity tree for experimental registrations, and 
4) the "Special X. Tree" for unregistered values. 
The MIME Registration Procedures, as outlined in Section 2.1.5 of RFC 
2048, has also established a provision for IANA to create a new top-
level registration tree, with the consent of the IESG, for the rare 
instances of registrations not fitting into one of these four 
categories.

Unfortunately, none of these four top-level registration trees are 
appropriate for identifying the codecs of "traditional" multimedia:
1) The IETF Tree can't be used because it would be prohibitively 
   time consuming to establish RFCs to identify each of the already 
   registered codecs.
2) The Vendor Tree can't be used because corporations other than the 
   Registrar usually developed these codecs. Appeals to these scores 
   of corporations to independently register their codecs is not 
   likely to result in uniform compliance and would also involve 
   significant overhead.
3) The Personal or Vanity tree is inappropriate because many of 
   these codecs represent serious, proven technology that has 
   endured through time.
4) The Special X. Tree is similarly inappropriate because each codec 
   identifier needs to be uniquely and publicly known within the 
   Internet namespace (i.e., IANA).

Therefore, a need exists by which codec identifiers, which have already 
been registered by non-IANA registries, may be re-registered within the 
IANA namespace. This document seeks to address this need by 
establishing a new top-level registration tree, the Codec Tree. 

2 Codec Tree Namespace

Codec identifiers within the codec tree take the form of audio/codec-
XXX for audio codecs and video/codec-XXX for video codecs (where the 
value "XXX" is the unique codec identifier within the original 
namespace). For example, an audio codec, which has the codec ID value 
of 123 within the WAVE registry, would have a name of "audio/codec-123" 
within the Codec Tree Namespace. Similarly, a video codec with a codec 
ID value of "SCRN" within the AVI registry would have a name of 
"video/codec-SCRN" within the Codec Tree Namespace. 

Because the Codec Tree namespace is potentially populated by entries 
coming from different registries, and because many codecs have been 
registered within several registries, the possibility exists that 
duplicate registrations for the same codecs may exist. For this reason, 
care should be taken, when registering new codec identifiers within the 


Fleischman                                                      [Page 3]

Internet-Draft                                           January 7, 1998

IANA Codec Tree namespace, to ensure that the codec identifier has not 
previously been registered within the namespace by a different 
Registrar. In this way duplicate registrations could hopefully be 
minimized.

3 WAVE and AVI Registries

Both the WAVE and AVI Registries are historic databases that have been 
maintained by Microsoft as a free service. The Registries sought to 
assist developers of WAVE and AVI content and to standardize WAVE and 
AVI content by
1) avoiding conflict and/or duplication with current definitions, and 
2) providing the registered information in a standard document and 
   format that is publicly available. 
The historic nature of these databases implies that unless the original 
registrants informed the registrar of a change of status (e.g., company 
acquired, new contact, new location, new phone), the contact 
information has generally not been updated from the originally 
registered values.

Audio codecs within the WAVE Registry are identified by WAVE Format 
IDs. The (audio) WAVE Format ID is officially known as "WAVE form 
Registration Number". The WAVE Format ID is a hexadecimal integer 
value. 

Video codecs within the AVI Registry are identified by AVI Codec IDs. 
The (video) AVI Codec ID value is a FourCC value. A FourCC is 32-bits 
long and represents a (case-sensitive) four-character (i.e., ASCII) 
code value. 

Direct mappings exist between WAVE Format IDs and GUIDs and between 
FourCC values and GUIDs [6]. [Note: GUIDs are Globally Unique 
Identifiers that are also known as Universally Unique Identifiers 
(UUIDs). UUIDs have been standardized within the Open Software 
Foundation's (OSF) Distributed Computing Environment (DCE).] These 
mappings enable GUID-oriented software to cleanly refer to these 
historic codec values. For example, the Advanced Streaming Format (ASF) 
[7] uses GUID values to refer to codecs, and this mechanism is used to 
convert the historic WAVE and AVI codec values into the appropriate 
GUID value.

WAVE Format IDs are converted to GUIDs by inserting the hexadecimal 
value of the WAVE Format ID into the XXXXXXXX part of the following 
template: {XXXXXXXX-0000-0010-8000-00AA00389B71}. For example, a WAVE 
Format ID of 123 has the GUID value of {00000123-0000-0010-8000-
00AA00389B71}.


Fleischman                                                      [Page 4]

Internet-Draft                                           January 7, 1998


FourCC values are converted to GUIDs by inserting the FourCC value into 
the XXXXXXXX part of the same template: {XXXXXXXX-0000-0010-8000-
00AA00389B71}. For example, a conversion of the FourCC value of "H260" 
would result in the GUID value of {30363248-0000-0010-8000-
00AA00389B71}. [Note: the 32-bit FourCC value of "H260" is converted 
into hexadecimal 32-bit value (i.e., 30363248) because the initial 
XXXXXXXX of the GUID is defined as a DWORD and thus takes a 32-bit 
hexadecimal value. Endian considerations account for the apparent re-
ordering of the original ASCII text.]
